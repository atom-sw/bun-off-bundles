# Explore, then code

Read before you write. Jumping straight to code produces solutions to the wrong problem.

- Understand the relevant files and the existing approach first. For anything spanning
  multiple files or an unfamiliar area, form a short plan before editing.
- Reuse what exists: prefer an existing function, utility, or pattern over a new one.
  Match the surrounding code's naming, structure, and idioms.
- Keep diffs minimal and focused on the task. Do not refactor unrelated code or restyle
  files you happen to touch.
- Skip the planning overhead for a change you could describe in one sentence (a typo, a
  log line, a rename): make it directly.
