# Writing good documentation

Documentation earns its place by saving the next reader time. Write it for that reader,
not for yourself.

- Document the intent and the non-obvious: why the code exists, the trade-offs, the
  constraints, and the gotchas. Do not restate what the code already says plainly.
- Keep documentation in sync with the code it describes. Update the docs in the same
  change as the behavior, and delete docs that no longer hold.
- Lead with the point: put the conclusion or the key instruction first, then the
  supporting detail.
- Prefer a concrete example or command over a paragraph of prose. Show the reader how.
- Structure the text for scanning: short sections, descriptive headings, and lists for
  steps or options.
